## 一、问题现状：传统轮询方式的Token浪费

- **背景：** 许多用户反馈使用 OpenClaw 调用 Claude Code 进行编程时，Token 消耗非常快。
- **原因：** 传统（常规）方式下，OpenClaw 需要通过"不断轮询"的方式来监控 Claude Code 的状态和输出。任务执行时间越长，轮询次数越多，消耗的 Token 就越多。
- **核心痛点：** OpenClaw 必须时刻"盯着"Claude Code，无法释放资源。

---

## 二、核心方案：Claude Code Hooks 与 Agent Teams

### Agent Teams 特性
这是 Claude Code 近期新增的强大功能，允许在内部创建一个完整的开发团队：
- **独立进程：** 每个 Agent 都是独立运行的，实现真正的并行执行。
- **协作机制：** Agent 之间可以相互通信、共享任务列表并进行专业角色分工（如前端、后端、测试等）。

### Hooks 功能
通过结合 Claude Code 的 Hook 功能，可以实现开发完成后自动回调，从而达成"零轮询"和极省 Token 的目标。

### 自动化效果
任务完成后，用户可以在聊天软件（如 Telegram/Discord）中收到包含任务描述、项目路径、耗时、文件结构等详细信息的通知。

---

## 三、零轮询工作流详解

1. **任务指派：** OpenClaw 将开发任务指派给 Claude Code。此操作仅执行一次，且在后台运行，不会阻塞 OpenClaw 的主对话窗口。

2. **自主开发：** Claude Code 接到任务后，独立进行开发与测试。此过程不消耗 OpenClaw 的上下文。

3. **触发 Hook：** 任务完成（触发 `stop` 事件）后，Claude Code 的 Hook 自动触发。
   - **双重保障：** 使用 `stop hook` 作为主回调，并辅以 `session end` 作为兜底，确保通知一定能发出。

4. **数据写入与唤醒：** Hook 将执行结果写入文件，并通过 Webhook 唤醒 OpenClaw。

5. **结果反馈：** OpenClaw 读取文件中的状态和结果，将其发送给用户。由于只在开始指派和结束读取时消耗 Token，总消耗几乎可以忽略不计（通常不超过 1000 字）。

## 四、案例演示：构建"落沙模拟游戏"

- **任务下达：** 用户在对话框输入指令，要求使用 Agent Team 模式构建一个基于物理引擎（HTML/CSS）带材质系统的落沙模拟游戏。

- **多任务并行：**
  - 指派任务后，主 Agent 进程未被阻塞，用户可以继续让它执行其他任务（如查询天气、讲笑话）。
  - Claude Code 在后台完全自主运行。

- **独立通知：** 为了不打乱主对话窗口的上下文，任务完成的消息被推送到专门的群组。

- **成果展示：**
  - 通知显示开发耗时约 6 分钟，通过了 184 个测试。
  - 用户请求打包文件并在浏览器中测试，游戏实现了沙子下落、水流、火焰上升、木头燃烧及蒸汽遇冷变雨等物理互动。

## 五、方案优势总结

- 实现了真正的全自动开发，Token 消耗极低
- 即使不在电脑前，也可以通过手机下达指令
- 主 Agent 进程不被阻塞，可以同时执行其他任务
- 任务完成后自动推送详细报告到群组
